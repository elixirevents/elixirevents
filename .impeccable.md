# ElixirEvents Design Context

## Users
Elixir and BEAM developers at all levels — from newcomers exploring the ecosystem to core team members and conference organizers. They visit to discover upcoming events, rewatch talks, find speakers, and browse topics. Contributors submit events and talks via YAML files and pull requests. The audience is global, technical, and community-oriented.

## Brand Personality
**Playful, Warm, Community-driven**

The "Elixir" name is taken literally — a magic potion. The brand leans into alchemy and wonder: a potion bottle logo with sparkles, purple elixir liquid, cork stopper. This isn't a corporate directory. It's a community project that should feel alive, fun, and welcoming. The tone is confident but never exclusive — every speaker and every event matters equally.

## Aesthetic Direction

### Visual Tone
Warm editorial meets magical playfulness. Think: a beautifully curated conference program designed by someone who loves the craft. Not cold, not corporate, not generic "developer tool" aesthetic.

### Primary Design Theme
**Dark theme is the guiding star.** Design dark-first, then adapt for light. The dark theme should feel rich and immersive — deep plum-tinted backgrounds with vibrant purple and coral accents. The light theme (warm cream) is a companion, not the primary experience.

### Brand Color
Primary purple: `#8a3897` / `oklch(45% 0.2 310)` — a warm magenta-purple, not a cold blue-purple.

### Color Palette
- **Primary**: `#8a3897` magenta-purple (dark: lighter variant for contrast)
- **Secondary**: Warm coral/orange (nod to the Elixir logo heritage)
- **Accent**: Teal/sage for status indicators and variety
- **Dark base**: Deep plum-tinted (`oklch(20% 0.025 285)`) — never pure black
- **Light base**: Warm cream (`oklch(97.5% 0.005 85)`) — never pure white
- **Neutrals**: Always tinted toward the brand hue

### Typography
- **Display**: Bricolage Grotesque (Google Fonts) — geometric, confident, personality
- **Body**: DM Sans (Google Fonts) — clean, readable, warm
- No monospace as lazy "developer" shorthand. The design should transcend "tech site."

### Signature Elements
- Potion bottle SVG logo with purple liquid and 4-pointed sparkle stars
- Sparkle decorations (4-pointed stars) as section accents — used sparingly
- Event cards with unique gradient backgrounds (each event has its own color identity)
- Initials-based speaker avatars with unique gradient colors (until real photos)
- Scroll-snap horizontal carousels for featured content

### Anti-References
- RubyEvents.org (functional inspiration but visually dated — our design should feel 5+ years more modern)
- Generic "AI slop" aesthetics: cyan-on-dark, purple-to-blue gradients, glassmorphism, bounce animations, gradient text
- Cold corporate dashboards
- Identical card grids with icon + heading + text repeated endlessly

### References (Feel, not copy)
- The warmth and editorial quality of well-designed conference programs
- The playfulness and personality of indie game sites
- The community energy of open source project homepages that feel alive

## Design Principles

1. **Events are the star.** Every design decision should make events easy to discover, explore, and revisit. Don't let chrome overshadow content.

2. **Inclusive by default.** No "featured speakers" pedestal. Show data (talk counts, recent activity) and let it speak. Copy should never imply hierarchy of people — only of data.

3. **Dark-first, both-excellent.** The dark theme is the primary experience. Design there first, then ensure the light theme is equally considered — not an afterthought.

4. **Fun is a feature.** The potion motif, sparkles, and playful touches aren't decoration — they're the brand. A directory doesn't have to be boring. Lean into the magic.

5. **Community over curation.** This is open source, PR-driven, community-maintained. Design should reflect openness: easy contribution paths, transparent data, no gatekeeping aesthetics.

## Accessibility
WCAG AAA where practical. Enhanced contrast ratios, reduced motion support via `prefers-reduced-motion`, keyboard navigation, semantic HTML, and screen reader compatibility. Extra care for cognitive load — clear hierarchy, consistent patterns, no overwhelming animations.

## Tech Stack
- Phoenix 1.8 + LiveView + Ecto + PostgreSQL
- Tailwind CSS v4 + daisyUI
- Google Fonts (Bricolage Grotesque, DM Sans)
- No JavaScript framework — server-rendered with LiveView for interactivity
